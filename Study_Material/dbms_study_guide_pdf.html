üéØ UNIT 1: Introduction to Database System
What is Data? (The Building Block)
Data is a collection of raw facts and figures that can be stored in various forms like text, numbers, images, or audio files.
Think of data like ingredients in your mom's kitchen. Rice grains, dal, vegetables - all individual items that don't mean much alone, but when combined properly, they become a delicious meal!
Your Aadhaar number (123456789012), your name (Yash), your photo - all are individual data pieces.
What is a Database?
A database is an organized collection of interrelated data that can be easily accessed, managed, and updated.
Imagine your WhatsApp as a database! It stores all your chats (data), organizes them by contacts (structure), lets you search messages (retrieval), and keeps everything safe with encryption (security). Every time you save a contact or send a message, you're doing database operations!
Real-Life Use Cases:
  • Banking: Your bank account info, transaction history
  • E-commerce: Flipkart's product catalog, your order history
  • Social Media: Instagram's photo storage, user profiles
  • Education: Your college's student records, attendance
Database Management System (DBMS)
DBMS is a software that provides an interface between users and databases to store, retrieve, and manage data while ensuring security and integrity.
DBMS is like FRIDAY (Tony Stark's AI). You ask FRIDAY to "show me all my suits" (query), and it knows exactly where to find them, how to display them, and who's allowed to access them. The suits are your data, the garage is your database, and FRIDAY is your DBMS!
Key Operations DBMS Performs:
  1. Data Definition - Creating tables (like setting up new rooms in your house)
  2. Data Manipulation - Adding, updating, deleting records
  3. Data Retrieval - Searching and displaying data
  4. User Administration - Managing who can access what

File system is like keeping your notes in different notebooks scattered around your room. DBMS is like having one organized digital notebook (like Notion) where everything is searchable and backed up!
DBMS Architecture - The Complete Picture

Data Abstraction Levels

Three-Tier Architecture:
  1. Physical Level - How data is actually stored (like the actual books in a library)
  2. Logical Level - What data is stored (like the library catalog)
  3. View Level - What users see (like the user-friendly search interface)
Real Example in IRCTC:
  • View Level: The booking page you see
  • Logical Level: All trains, routes, seat availability data
  • Physical Level: Servers storing this data in Mumbai, Delhi data centers
Two-Tier and Three-Tier Database Architecture

Two-Tier Architecture (2-Tier)

In two-tier architecture, the application (client) communicates directly with the database (server). The client sends SQL queries to the database and gets back results, with NO extra layer in between.
Think of it like ordering food via a phone call directly to the restaurant. You (client) call the restaurant (server) and order food. The restaurant cooks and gives you the dish.
Small desktop apps where the user and database are on the same local network, Old banking software running on office PCs

Three-Tier Architecture (3-Tier)

In three-tier architecture, there's an extra layer‚Äîthe application server‚Äîbetween the client and the database. The client talks to the application server, which then talks to the database server.
It's like ordering food on Zomato‚ÄîYou (client) use the Zomato app (application server), which sends orders to the restaurant (database server), and then food is delivered to you!
Modern web apps (like IRCTC, Amazon) where your browser talks to a web server, which fetches data from backend database servers
Layer/Role 2-Tier 3-Tier
Client Directly talks to DB server Talks to Application/Business Logic Layer
Application code Often mixed with user interface In separate server (middle layer)
Best for Small network apps Large, scalable web/mobile apps
DBMS Functional Components
  • Storage Manager: Handles physical storage and retrieval of data; controls file, buffer, authorization, transaction
  • Query Processor: Translates, optimizes, and executes queries you write (e.g. SELECT ...)
Types of DBMS Users
  • Naive users: Use application interface (e.g. ATM users, ordering on Zomato)
  • Application Programmers: Write apps that use the DB
  • Sophisticated users: Use advanced query tools (analysts)
  • DBA (Database Admin): Controls permissions, optimizes and secures DB
Data Independence
Physical Data Independence: "Ability to change physical storage without affecting logical structure."
Logical Data Independence: "Ability to change logical schema without changing user views or application programs."
Data Models - Different Ways to Think About Data
  1. Hierarchical Model: Think of it like a family tree - one parent, many children, but each child has only one parent.
  2. Network Model: Like your social media connections - more complex relationships.
  3. Relational Model (Most Important!): Data stored in tables with rows and columns - like Excel sheets that are connected to each other.
Complete Codd's Rules (All 13 Rules)

Dr. E.F. Codd gave us 13 rules for relational databases. Here are ALL of them:

Rule 0: Foundation Rule
"For a system to qualify as RDBMS, it must be able to manage databases entirely through its relational capabilities."
Rule 1: Information Rule
"All information in the database must be represented as values in tables (cells of rows and columns)."
Rule 2: Guaranteed Access Rule
"Every data value must be accessible using table name, primary key value, and column name."
Rule 3: Systematic Treatment of NULL Values
"NULL values must be handled uniformly and systematically, independent of data type."
Rule 4: Active Online Catalog
"Database catalog must be stored online and accessible using the same query language used for data."
Rule 5: Comprehensive Data Sublanguage Rule
"There must be at least one language that supports data definition, manipulation, and transaction management."
Rule 6: View Updating Rule
"All views that are theoretically updatable must be updatable by the system."
Rule 7: High-level Insert, Update, Delete
"The system must support high-level insert, update, and delete operations on sets of data."
Rule 8: Physical Data Independence
"Changes to physical storage must not affect application programs."
Rule 9: Logical Data Independence
"Changes to logical structure must not affect application programs."
Rule 10: Integrity Independence
"Integrity constraints must be stored in catalog and enforced automatically."
Rule 11: Distribution Independence
"Distribution of data across locations must be invisible to users."
Rule 12: Non-Subversion Rule
"Low-level access must not bypass security and integrity constraints."

üîó UNIT 2: Relational Data Model
Relational Structure - The Foundation

Tables/Relations:

A table is a collection of related data organized in rows and columns.
Think of a table like your class attendance sheet. Each row is one student, each column is information about that student (roll no, name, attendance).
Key Terms:
  • Tuple/Row: One complete record (one student's info)
  • Attribute/Column: One piece of information (like "Name" column)
  • Domain: Allowed values for an attribute (like marks can only be 0-100)
Keys - The Complete Guide

1. Super Key

"A set of one or more attributes that can uniquely identify any tuple in a relation."
Like having multiple ways to identify you - your roll number alone, or your name + phone number together.
In Student(RollNo, Name, Email, Phone), super keys are:
  • {RollNo}
  • {Email}
  • {RollNo, Name}
  • {Email, Phone}

2. Candidate Key

"A minimal super key with no redundant attributes."
From above super keys, candidate keys are:
  • {RollNo}
  • {Email}

3. Primary Key

"A candidate key chosen to uniquely identify tuples, cannot be NULL."
Think of Primary Key as your superhero identity number. Spider-Man has only one unique ID in the Marvel universe, just like each student has only one unique roll number.
Rules for Primary Key:
  • Must be unique (no duplicates)
  • Cannot be NULL
  • Value should not change
  • Table can have only ONE primary key

4. Foreign Key

"An attribute in one table that refers to the primary key of another table."
In a college database:
  • Student table: StudentID (Primary Key), Name, Email
  • Enrollment table: StudentID (Foreign Key), CourseCode, Grade
Data Constraints - The Rules

Why Constraints? To maintain data quality and prevent garbage data entry.

Types of Constraints:

1. NOT NULL Constraint

Name VARCHAR(50) NOT NULL

Meaning: This field cannot be empty (like mandatory fields in a form)

2. UNIQUE Constraint

Email VARCHAR(100) UNIQUE

Meaning: No two records can have the same email

3. CHECK Constraint

Age INT CHECK(Age >= 18)

Meaning: Age must be 18 or above (like age verification for voting)

4. DEFAULT Constraint

Country VARCHAR(50) DEFAULT 'India'

Meaning: If no country is specified, automatically set to 'India'

5. PRIMARY KEY Constraint

StudentID INT PRIMARY KEY

6. FOREIGN KEY Constraint

FOREIGN KEY (StudentID) REFERENCES Student(StudentID)
Functional Dependencies - The Logic
If for every value of attribute A, there's exactly one value of attribute B, then B is functionally dependent on A (A ‚Üí B).
  • StudentID ‚Üí StudentName (One student ID always maps to one name)
  • StudentID ‚Üí Email (One student ID always maps to one email)
  • But NOT: StudentName ‚Üí StudentID (multiple students might have same name)
Normalization - Data Organization

Why Normalize? To reduce redundancy and maintain data integrity.

First Normal Form (1NF):

Each cell should contain only one value (atomic values)

Second Normal Form (2NF):

Must be in 1NF + all non-key attributes fully depend on primary key

Third Normal Form (3NF):

Must be in 2NF + no transitive dependencies
Student database:
  • Before 3NF: StudentID, Name, DeptID, DeptName, DeptHead
  • After 3NF:
    • Student(StudentID, Name, DeptID)
    • Department(DeptID, DeptName, DeptHead)
Entity-Relationship (ER) Model - Complete Guide
Components:

1. Strong Entity

Symbol: Rectangle

Student, Course, Teacher

2. Weak Entity

Symbol: Double Rectangle

Dependents of employees (can't exist without employee)

3. Attributes

Symbol: Oval

Types:
  • Key Attribute: Underlined oval (Primary Key)
  • Composite Attribute: Multiple ovals connected to main oval
  • Multi-valued Attribute: Double oval
  • Derived Attribute: Dashed oval

4. Relationships

Symbol: Diamond

Cardinality Types:
  • One-to-One (1:1): One student has one ID card
  • One-to-Many (1:M): One teacher teaches many students
  • Many-to-Many (M:N): Many students enrolled in many courses
Cardinality Notations:
  • One-to-One (1:1): Single line with 1 on both sides
  • One-to-Many (1:M): Single line on one side, crow's foot on other
  • Many-to-Many (M:N): Crow's foot on both sides

üíª UNIT 3: SQL and NoSQL
SQL - Structured Query Language
SQL is a standard language used to interact with relational databases for creating, retrieving, updating, and managing data.
SQL is like English for talking to databases! Instead of "Please show me all students," you write "SELECT * FROM Students."
Components of SQL

1. DDL (Data Definition Language)

Purpose: Define database structure

Commands:
  • CREATE - Make new tables/databases
  • ALTER - Modify table structure
  • DROP - Delete tables/databases
  • TRUNCATE - Remove all data but keep structure

2. DML (Data Manipulation Language)

Purpose: Manipulate data

Commands:
  • INSERT - Add new records
  • UPDATE - Modify existing data
  • DELETE - Remove records
  • SELECT - Retrieve data

3. DCL (Data Control Language)

Purpose: Control access permissions

Commands:
  • GRANT - Give permissions
  • REVOKE - Remove permissions

4. TCL (Transaction Control Language)

Purpose: Manage transactions

Commands:
  • COMMIT - Save changes permanently
  • ROLLBACK - Undo changes
  • SAVEPOINT - Create checkpoint
Complete SQL Data Types
Numeric Types:
  • TINYINT - Very small integers (0 to 255)
  • SMALLINT - Small integers (-32,768 to 32,767)
  • INT - Standard integers
  • BIGINT - Large integers
  • DECIMAL(p,d) - Fixed-point numbers
  • FLOAT - Floating-point numbers
String Types:
  • CHAR(n) - Fixed-length strings
  • VARCHAR(n) - Variable-length strings
  • TEXT - Large text data
Date/Time Types:
  • DATE - Date values (YYYY-MM-DD)
  • TIME - Time values (HH:MM:SS)
  • DATETIME - Date and time together
Basic SQL Structure
SELECT column1, column2 FROM table_name WHERE condition GROUP BY column HAVING condition ORDER BY column;
SQL Clauses

SELECT Clause:

SELECT Name, Marks FROM Students; SELECT * FROM Students; -- All columns

WHERE Clause:

SELECT * FROM Students WHERE Marks > 80; SELECT * FROM Students WHERE Name = 'Yash' AND Marks > 75;

GROUP BY Clause:

SELECT Department, COUNT(*) FROM Students GROUP BY Department;

HAVING Clause:

SELECT Department, COUNT(*) FROM Students GROUP BY Department HAVING COUNT(*) > 5;

ORDER BY Clause:

SELECT * FROM Students ORDER BY Marks DESC; SELECT * FROM Students ORDER BY Name ASC;

SQL Joins - Connecting Tables

INNER JOIN:

SELECT Students.Name, Courses.CourseName FROM Students INNER JOIN Enrollment ON Students.ID = Enrollment.StudentID INNER JOIN Courses ON Enrollment.CourseID = Courses.ID;
INNER JOIN is like finding common friends between you and someone else on Facebook - only shows people who are friends with both of you!
Types of Joins:
  • LEFT JOIN: All records from left table + matching from right
  • RIGHT JOIN: All records from right table + matching from left
  • FULL JOIN: All records from both tables

NoSQL - Not Only SQL
Why NoSQL?
Traditional SQL databases struggle with:
  • Big Data - Massive volumes of data
  • Variety - Different data formats (text, images, videos)
  • Velocity - High-speed data processing
  • Flexibility - Changing data requirements
Types of NoSQL Databases

1. Document Databases

Example: MongoDB

Use Case: Content management, catalogs

Structure: Stores data as documents (like JSON files)

2. Key-Value Stores

Example: Redis, DynamoDB

Use Case: Caching, session management

Structure: Simple key-value pairs

3. Column-Family

Example: Cassandra

Use Case: Time-series data, IoT data

Structure: Data stored in column families

4. Graph Databases

Example: Neo4j

Use Case: Social networks, recommendation engines

Structure: Nodes and relationships
